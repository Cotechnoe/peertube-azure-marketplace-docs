# Basic Administration

This page covers the essential operations for managing your PeerTube instance on Azure.

## Service management

PeerTube runs as a systemd service. Use these commands to control it:

```bash
# Check status
sudo systemctl status peertube.service

# Start
sudo systemctl start peertube.service

# Stop
sudo systemctl stop peertube.service

# Restart
sudo systemctl restart peertube.service

# View logs (last 50 lines)
sudo journalctl -u peertube.service -n 50

# Follow logs in real time
sudo journalctl -u peertube.service -f
```

## Related services

PeerTube depends on three services that are managed independently:

| Service | Check status |
|---------|-------------|
| nginx (reverse proxy) | `sudo systemctl status nginx` |
| PostgreSQL (database) | `sudo systemctl status postgresql` |
| Redis (cache) | `sudo systemctl status redis-server` |

All three must be running for PeerTube to operate correctly.

## Configuration file

The main PeerTube configuration file is located at:

```
/var/www/peertube/versions/peertube/config/production.yaml
```

After editing this file, restart PeerTube for changes to take effect:

```bash
sudo systemctl restart peertube.service
```

> Do not edit this file while PeerTube is processing uploads or transcoding videos.

## Retrieving passwords

All secrets are generated on the VM at first boot — they are never stored in the image.

### PeerTube administrator password (`root`)

Generated by PeerTube itself on first startup. Retrieve it from the system journal:

```bash
sudo journalctl -u peertube.service --no-pager | grep "User password"
```

Example output:
```
Jun 22 08:00:01 my-vm node[1234]: User password: AbCdEf123456
```

This password is used to log in to the web interface as `root`.

### PostgreSQL password

Written to the PeerTube configuration file:

```bash
sudo grep 'password' /var/www/peertube/versions/peertube/config/production.yaml
```

Or read the value directly from the `database.password` field:

```bash
sudo python3 -c "import yaml; cfg=yaml.safe_load(open('/var/www/peertube/versions/peertube/config/production.yaml')); print(cfg['database']['password'])"
```

### PeerTube application secret

Written to the same file under `secrets.peertube`:

```bash
sudo python3 -c "import yaml; cfg=yaml.safe_load(open('/var/www/peertube/versions/peertube/config/production.yaml')); print(cfg['secrets']['peertube'])"
```

### If passwords were provided via Custom data

If you supplied `peertube_db_password` or `peertube_secret` in the **Custom data** field when creating the VM, those values were used and are also readable from `production.yaml` as shown above.

### Reset the PeerTube administrator password

If you have lost the `root` password, reset it using the PeerTube CLI:

```bash
cd /var/www/peertube/versions/peertube
sudo -u peertube NODE_CONFIG_DIR=/var/www/peertube/versions/peertube/config \
  NODE_ENV=production node dist/server/tools/peertube-auth.js reset-password \
  -u root -p NewPassword
```

## Disk usage

Video storage is located under:

```
/var/www/peertube/storage/
```

Monitor disk usage with:

```bash
df -h /
du -sh /var/www/peertube/storage/
```

The OS disk is provisioned at 30 GB by default. If you plan to host large volumes of video, attach a separate data disk and configure PeerTube to use it via the `storage` section in `production.yaml`.

## Update PeerTube

Updating PeerTube requires manual steps. Follow the official upgrade guide:

> [https://docs.joinpeertube.org/maintain/upgrade](https://docs.joinpeertube.org/maintain/upgrade)

Always back up your database and storage before upgrading.

## Backup

### Database backup

```bash
sudo -u postgres pg_dump peertube_prod > peertube_backup_$(date +%Y%m%d).sql
```

### Storage backup

```bash
sudo tar czf peertube_storage_$(date +%Y%m%d).tgz /var/www/peertube/storage/
```

Store backups off the VM (Azure Blob Storage, local workstation, etc.).

## Nginx TLS certificate renewal

If you used **Let's Encrypt**, certificate renewal is handled automatically by the certbot systemd timer. Verify it is active:

```bash
sudo systemctl status certbot.timer
```

If you used the **self-signed certificate** generated during finalization, it is valid for 30 days. Re-run the finalization script to regenerate it, or replace it with a Let's Encrypt certificate by providing a domain name.

## Firewall / NSG

The Azure Network Security Group (NSG) controls inbound traffic. Recommended rules after deployment:

| Port | Protocol | Purpose | Restriction |
|------|----------|---------|-------------|
| 22 | TCP | SSH administration | Restrict to your IP |
| 80 | TCP | HTTP (redirect to HTTPS) | Open |
| 443 | TCP | HTTPS (PeerTube) | Open |

Adjust NSG rules in the Azure Portal under **Networking** on your VM's page.
